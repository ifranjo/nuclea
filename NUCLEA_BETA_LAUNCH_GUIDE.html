2. Supabase Cloud

Supabase Cloud reemplaza el Docker local. Todos los datos se almacenan en la UE (Frankfurt) para cumplimiento RGPD.

Pasos

  1. 1 Ir a supabase.com/dashboard y crear un nuevo proyecto.
  2. 2 Región: seleccionar eu-west-1 (Frankfurt) para cumplimiento RGPD.
  3. 3 Apuntar el project ref (visible en la URL: supabase.com/dashboard/project/<ref>).
  4. 4 Guardar desde Settings > API: Project URL, anon key y service role key.

Vincular y subir migraciones

cd POC_REAL npx supabase login npx supabase link --project-ref <tu-project-ref> npx supabase db push

Sobre RLS: La migración 00002 desactiva RLS para desarrollo local. La migración 00012 reactiva RLS en las 12 tablas con políticas de producción. Como las migraciones se ejecutan en orden, el estado final siempre tiene RLS activo.

Verificar en el dashboard

  • Storage: El bucket capsule-contents debe existir con public = false.
  • Auth > Providers: Email habilitado (por defecto).
  • Auth > URL Configuration:
    • Site URL: https://tu-dominio.vercel.app
    • Redirect URLs: https://tu-dominio.vercel.app/** y http://localhost:3002/**

Pasos

  2. 2 Root Directory: establecer como POC_REAL.
  3. 3 Framework Preset: Next.js (auto-detected).
  4. 4 Build Command: next build (por defecto).
  5. 5 Install Command: npm ci (por defecto).
  6. 6 Añadir todas las variables de entorno (ver sección 5).
  7. 7 Deploy desde la rama master.

4. Email (Resend)

NUCLEA necesita enviar emails transaccionales: invitaciones beta, notificaciones de expiración, avisos a personas de confianza. El SDK de Resend ya está instalado en package.json (resend ^6.9.2).

Pasos de configuración

  1. 1 Crear cuenta en resend.com.
  2. 2 Añadir dominio nuclea.app en Settings > Domains.
  3. 3 Configurar registros DNS: SPF, DKIM, DMARC (Resend proporciona los valores).
  4. 4 Generar API key en Settings > API Keys.
  5. 5 Añadir RESEND_API_KEY y EMAIL_FROM=NUCLEA <noreply@nuclea.app> a Vercel.
  6. 6 Verificar envío con un test: curl -X POST https://tu-dominio.vercel.app/api/beta/invite.

Emails del sistema: Supabase Cloud usa su propio servicio para los emails de autenticación (OTP, magic links). Resend se usa solo para emails transaccionales de la aplicación (invitaciones, notificaciones). Son dos canales distintos.

6. Beta Gate

El sistema de beta gate controla quién puede acceder a la aplicación. Solo los usuarios con acceso beta concedido pueden usar las rutas protegidas.

Cómo funciona

  1. Variable de control: BETA_GATE_ENABLED=true en las variables de entorno. Si es false o no existe, todas las rutas son accesibles (modo desarrollo).
  2. Middleware: El archivo src/middleware.ts intercepta cada petición:
    • Rutas públicas (/login, /registro, /onboarding, /share/*, /beta/*): acceso libre.
    • Rutas protegidas (/dashboard, /capsule/*, /settings): requieren autenticación + acceso beta.
  3. Verificación: El middleware busca en users el auth_id del usuario autenticado, luego comprueba si tiene un registro en beta_access con enabled = true.
  4. Redirección: Si no tiene acceso beta, redirige a /beta/waitlist (página de espera).
  5. Fail closed: Si hay cualquier error de base de datos durante la verificación, se redirige a waitlist (nunca se concede acceso por error).

Flujo de invitación beta

PasoAcciónEndpoint
1Admin envía invitación por emailPOST /api/beta/invite
2Usuario recibe enlace con token/beta/accept?t=<token>
3Usuario introduce su email, recibe OTPPOST /api/beta/accept
4Verifica OTP, queda autenticadoSupabase Auth (magic link)
5Se concede acceso betaPOST /api/beta/complete
6Accede al dashboard/dashboard

Seguridad del token

  • Generado con crypto.randomBytes(32) (256 bits).
  • Solo se almacena el hash SHA-256 en la tabla beta_invitations (nunca el token en claro).
  • Rate limit: 5 intentos cada 15 minutos por IP (tabla beta_rate_limits).
  • Cada token tiene fecha de expiración configurable.

7. Verificación Post-Deploy

Ejecutar estas 9 comprobaciones después de cada despliegue. Sustituir TU_DOMINIO por el dominio real de Vercel.

7.1. Auth callback

Verificar que la URL de redirección de Supabase coincide con el dominio de Vercel.

# En Supabase Dashboard > Authentication > URL Configuration # Site URL debe ser: https://TU_DOMINIO.vercel.app # Redirect URLs debe incluir: https://TU_DOMINIO.vercel.app/**

7.2. RLS activo

Ejecutar en Supabase SQL Editor:

SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname = 'public' ORDER BY tablename;

Todas las tablas deben mostrar rowsecurity = true.

7.3. Storage privado

En Supabase Dashboard > Storage, verificar que el bucket capsule-contents tiene public = false.

7.4. Beta gate funcional

# Acceder sin sesión beta — debe redirigir a /beta/waitlist curl -sI https://TU_DOMINIO.vercel.app/dashboard | grep -i location

7.5. Cron endpoint

curl -X POST https://TU_DOMINIO.vercel.app/api/cron/lifecycle \ -H "x-cron-secret: TU_CRON_SECRET" \ -w "\nHTTP Status: %{http_code}\n"

Debe devolver 200.

7.6. Registro de usuario

Crear una cuenta manualmente en /registro. Confirmar email. Verificar que aparece un registro en la tabla users.

7.7. Invitación beta

curl -X POST https://TU_DOMINIO.vercel.app/api/beta/invite \ -H "Content-Type: application/json" \ -H "x-admin-key: TU_ADMIN_API_SECRET" \ -d '{"email": "test@example.com", "cohort": "c1-launch"}' \ -w "\nHTTP Status: %{http_code}\n"

Debe devolver 200 con el token de invitación.

7.8. Upload de archivos

Iniciar sesión, crear una cápsula, subir una imagen. Verificar en Supabase Storage que aparece bajo capsule-contents/{capsule_id}/.

7.9. Página de compartir

# Abrir en modo incógnito (sin sesión) # Debe renderizar la cápsula en solo lectura curl -sI https://TU_DOMINIO.vercel.app/share/TOKEN_DE_PRUEBA | grep -i "200\|301\|302"

8. Admin Panel

La interfaz de administración está en /admin/beta. Requiere autenticación con la API key de admin.

Acceso

  1. Navegar a https://TU_DOMINIO.vercel.app/admin.
  2. Introducir la ADMIN_API_SECRET cuando se solicite.
  3. Se redirige automáticamente a /admin/beta.

Funcionalidades

PestañaFuncionalidad
Invitaciones Enviar nuevas invitaciones por email, ver estado (pendiente, aceptada, expirada, revocada), reenviar invitación
Acceso Ver usuarios con acceso beta (nombre, email, cohorte, fecha), revocar acceso individual
Auditoría Log cronológico de todos los eventos: invitación enviada, aceptada, completada, revocada. Incluye IP y user agent

Endpoints API del admin

MétodoEndpointHeaderAcción
POST/api/beta/invitex-admin-keyCrear invitación
GET/api/beta/invitex-admin-keyListar invitaciones
POST/api/beta/invite/resendx-admin-keyReenviar invitación
POST/api/beta/invite/revokex-admin-keyRevocar invitación
POST/api/beta/revokex-admin-keyRevocar acceso beta
GET/api/beta/accessx-admin-keyListar accesos
GET/api/beta/auditx-admin-keyLog de auditoría

Autenticación admin: Todas las llamadas API requieren el header x-admin-key con el valor de ADMIN_API_SECRET. El admin panel envía este header automáticamente después de autenticarse.
